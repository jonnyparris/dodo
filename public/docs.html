Architecture

Every Dodo deployment runs as a Cloudflare Worker with two types of Durable Objects: one per user (state, config, tasks) and one per session (code, chat, git).

Life of a request

Session Lifecycle

Sessions can be soft-deleted via DELETE /session/:id and recovered within 5 minutes using POST /session/:id/restore. After 5 minutes, deleted sessions are permanently purged.

Use POST /api/sessions/:id/fork to copy a session's files, git history, and conversation into a new session.

Sessions

Each session is an isolated workspace with its own files, git repo, message history, and token budget. Sessions run in separate Durable Objects, so they never interfere with each other.

  • Create — starts with an empty workspace. Clone a repo or let the agent build from scratch.
  • Fork — copies all files and the full message history into a new session. Branch an experiment without losing the original.
  • Rename — click the session title in the header bar to rename it.
  • Delete — permanently removes the session and all its data.

Chat & Streaming

The web UI uses the prompt path and streams responses into the chat over SSE as the agent works.

ModeButtonHow it works
SendSendStarts a prompt, returns immediately, and streams output into the open chat over SSE. The input is disabled while the agent works.
Prompt RecoveryAutomaticPrompts are fiber-backed. If the Durable Object is evicted or the page reconnects, Dodo resumes and finishes the prompt instead of starting over.

Prompts are fiber-backed: if the Durable Object is evicted mid-prompt, it replays from a checkpoint on recovery rather than re-running the full LLM call.

  • Abort — cancels a running prompt immediately.
  • Streaming renderer — text appears immediately as deltas arrive, then Dodo upgrades it into full markdown a few times per second so long messages stay responsive.
  • Markdown — assistant messages render GitHub-flavored markdown. Each code block has a copy button.
  • Presence — when multiple users share a session, you see avatar dots in the header and typing indicators.
  • Build marker — the header shows the deployed build hash when available, so you can confirm which revision the UI is running.

Workspace Files

Each session has a virtual filesystem backed by SQLite (via @cloudflare/shell). The agent reads and writes files autonomously during conversations.

  • Click a directory to expand it. Click a file to view its contents in a modal.
  • The file tree refreshes automatically as the agent creates, edits, or deletes files.
  • Full CRUD, glob search, and in-file search/replace are available to the agent via its built-in tools.

Git

Full git support via isomorphic-git, running inside the Durable Object. The agent can clone repos, make commits, push and pull — all without a local machine.

OperationDetails
CloneShallow by default (depth 1). Supports GitHub and GitLab. Tokens are injected from your secrets.
Add / CommitStage files and commit with a message. The agent does this autonomously.
Branch / CheckoutCreate and switch branches.
Push / PullPush to and pull from remote repos. Auth tokens from your stored secrets are injected automatically.
Diff / Log / StatusView changes, history, and working tree status.

Store your GitHub or GitLab token in Secrets (sidebar). The agent injects them into git operations automatically — you never paste tokens into chat.

Task Board (Kanban)

The task board is a mini Kanban in the right sidebar. Add tasks, organize them across four columns, and dispatch them to run as autonomous sessions.

Columns

ColumnMeaning
BacklogTasks you've captured but haven't committed to yet.
TodoReady to work on. The Dispatch button appears here.
In ProgressA session is actively working on this task. Click the session link to watch.
DoneCompleted tasks. Delete when you no longer need them.

Card actions

Each card shows contextual buttons depending on its column:

ColumnAvailable actions

Todo▶ Dispatch (create session + run), ← (back to backlog), × (delete)
In Progress✓ Done (mark complete), 💻 session (navigate to session), × (cancel)
Done× (delete)

Dispatch

Dispatching a task creates a brand new session and sends the task title, description, and priority as an async prompt. The agent starts working immediately. The UI navigates you to the new session so you can watch.

A confirmation dialog appears before dispatch. Dispatching creates a session and starts an LLM call — it won't happen from an accidental click.

Drag and drop

Drag cards between columns to change their status. Dropping a card into In Progress triggers dispatch (with a confirmation).

Multi-select and batch operations

Hold Ctrl (or Cmd on Mac) and click cards to select multiple tasks. A batch bar appears above the board with Dispatch N and Delete N buttons. Batch dispatch creates parallel sessions — one per task.

Auto-sync

Every 30 seconds, Dodo checks whether dispatched sessions have finished. If a session's prompt completed, the task moves to Done automatically. If it failed, it moves back to Backlog.

Priority

Tasks have three priority levels — low, medium, high — set when creating a task. Priority is included in the dispatch prompt and color-coded on the card (red for high, green for low).

Memory

A persistent key-value store that survives across sessions. Use it to store patterns, API keys, project context, or anything the agent should remember.

  • Each entry has a title, content body, and tags.
  • Search by keyword. The agent can also read and write memory via its tools.
  • Memory is scoped to your user account, not individual sessions.

Cron Jobs

Schedule prompts that run automatically. Each cron job sends a prompt to its session after a delay.

FieldDescription
DescriptionA label for the job (shown in the list).
PromptThe message sent to the agent when the job fires.
Delay (seconds)How long to wait before firing.

Cron jobs run via the Agents SDK scheduler inside the Durable Object. They persist across DO evictions.

Integrations (MCP Servers)

Extend the agent's capabilities by connecting external MCP servers. When enabled, their tools become available to the agent during conversations.

  • Add — provide a name, URL, and optional auth header. Dodo connects over HTTP.
  • Toggle — enable or disable per integration without deleting the config.
  • Test — verifies the connection and reports how many tools are available.
  • Per-session overrides — in Session Settings, you can enable/disable integrations for a specific session without changing the global config.

Session Settings

Click Settings in the header bar to open the session settings overlay.

SectionWhat it does
SharingCreate share links with read-only or read-write access. Revoke links at any time.
PermissionsGrant other users direct access by email address. Choose read-only or read-write.
Browser ToolsToggle browser capabilities for the session (when available).
ApprovalsReview and approve/reject pending tool calls that require human authorization.
MCP OverridesEnable or disable integrations for this session only. Reset to follow the global config.

Config & Secrets

Config

Set your LLM model and gateway in the left sidebar. Changes apply to all new prompts.

SettingDescription
ModelThe LLM model ID (e.g. anthropic/claude-sonnet-4-5). Populated from your gateway.
GatewayAI Gateway (Cloudflare) or OpenCode. Determines which endpoint receives LLM requests.
Gateway URLThe base URL of your gateway.

Secrets

Tokens are encrypted at rest using a passkey you set. Dodo never stores plaintext credentials.

  • Set a passkey first (min 4 characters). This unlocks the encrypted secret store.
  • Store GitHub, GitLab, Gateway, and ntfy tokens.
  • Tokens are injected into git operations and LLM calls automatically.

Allowlist

When the agent runs sandboxed code, it can only reach hosts on the allowlist. Add hostnames (e.g. api.github.com) to grant network access. Known MCP server hosts are allowed automatically.

Connecting via MCP

Dodo exposes an MCP server at /mcp. Connect from OpenCode, Claude Desktop, or any MCP client to create sessions, send prompts, manage tasks, and more — all from your terminal.

OpenCode config

{
  "mcp": {
    "dodo": {
      "type": "remote",
      "url": "https://your-dodo.workers.dev/mcp",
      "headers": {
        "Authorization": "Bearer {file:.opencode/dodo-mcp-token}"
      }
    }
  }
}

Setup

  1. Generate a token: openssl rand -base64url 32 > ~/.opencode/dodo-mcp-token
  2. Set it as a Worker secret: wrangler versions secret put DODO_MCP_TOKEN
